# Shared partials (header & footer)

The marketing site is a no-build static template, so there is no server-side
include. To keep **one source of truth** for the site menu and footer, the
canonical markup lives in [`partials/`](../partials/) and is stamped into each
page between HTML-comment markers by a small tool.

| Partial | Source of truth | Lifted originally from |
|---------|-----------------|------------------------|
| [`partials/header.html`](../partials/header.html) | the main menu | `index.html` |
| [`partials/footer.html`](../partials/footer.html) | the `footer-four` footer | `services.html` |
| [`partials/header.en.html`](../partials/header.en.html) | the English menu (`en/` pages) | hand-authored |
| [`partials/footer.en.html`](../partials/footer.en.html) | the English footer (`en/` pages) | hand-authored |

Pages under `en/` are stamped with the `.en` variants; everything else gets
the Greek ones. The English partials are hand-authored translations — the
sync only reads them, never bootstraps or overwrites them. Keep the footer's
`RealEstateAgent` JSON-LD **byte-identical** across both footers (same
`@id …/#organization`). The header's language switcher (`.fw-lang-switch`)
ships a static `/` ↔ `/en/` fallback; `js/fourwalls.js` retargets it to the
exact alternate page from the head's hreflang links at runtime.

## How it works

Each participating page has the block wrapped in markers:

```html
<!-- FW:INCLUDE header — generated by tools/sync-partials.js; edit partials/header.html -->
	…canonical header…
<!-- /FW:INCLUDE header -->
```

[`tools/sync-partials.js`](../tools/sync-partials.js) copies each partial into
the region between its markers, on every listed page.

Because the markup is **physically present in every page** (not injected at
runtime), the theme JS keeps working exactly as before — sticky menu, Bootstrap
dropdowns, the footer's `lazy-img` shape — with **no flash on load**. The
per-page "current page" menu highlight is applied at runtime by
[`js/fourwalls.js`](../js/fourwalls.js) (`markCurrentPage`), so the shared menu
needs no per-page edits.

## Workflow

1. Edit [`partials/header.html`](../partials/header.html) or
   [`partials/footer.html`](../partials/footer.html) — **never** edit the block
   inside a page (a sync overwrites it).
2. Run the sync:

   ```bash
   node tools/sync-partials.js
   ```

3. Hard-refresh the preview (**Ctrl+F5**).

## Which pages are covered

The `PAGES` array at the top of [`tools/sync-partials.js`](../tools/sync-partials.js)
lists them — all the Greek-adapted pages (`index.html`, `services.html`,
`services/*.html`, `properties.html`, `property.html`, `about.html`,
`contact.html`, the legal pages, `404.html`, …).

**To add a page:** put its filename in `PAGES` and run the sync — the tool finds
the page's existing `<header class="theme-main-menu …">` / `<div class="footer-…">`
region, wraps it in markers, and replaces it with the canonical partial.

Notes / limits:

- The partials' internal links are root-absolute and their asset paths
  relative (`images/logo/…`), so pages at the repo root resolve them as-is.
  A page in a subfolder (e.g. `services/*.html`, or the `property.html`
  shell served at `/properties/<code>`) carries `<base href="/">` in its
  `<head>` so those relative asset paths still resolve from the site root.
- First run bootstraps the two partials from their source pages; it **never**
  overwrites a partial that already exists.
